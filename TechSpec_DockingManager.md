# YMM4 Docking Manager 開発技術定義書

## 1. プロジェクト概要
任意の外部アプリケーションのウィンドウをYMM4（ゆっくりムービーメーカー4）の内部にUIパネルとしてアタッチ（ドッキング）させ、一元管理するためのプラグイン「Docking Manager」を開発する。最大10個のドッキングパネルを生成・管理し、アタッチされたアプリケーションの状態を保存して次回起動時に自動復元する機能を備える。

## 2. 求められる主要機能
1. **ドッキングUI管理画面 (メインパネル):**
   - どのアプリケーションがどこのパネルにアタッチされているかを一覧表示。
   - 最大10個のアタッチ用ドッキングUI（サブパネル）の管理。
   - アタッチの解除、表示/非表示の切り替えなどを一元管理する。
2. **アタッチ用ドッキングUI (サブパネル 最大10個):**
   - アタッチされた外部アプリを表示する領域。ボタンなどの不純なUIは持たせず、領域全体を外部アプリの表示画面として使用する。
3. **Ctrl + ドラッグ＆ドロップによるアタッチ:**
   - 外部アプリのタイトルバーをCtrlキーを押下しながらサブパネル内にドロップした際にアタッチを受け付ける（誤爆防止仕様）。
4. **自動復元 (アタッチ状態の保存と復帰):**
   - パネルごとにどのアプリ・プロセス（実行ファイルパスや識別子）がアタッチされたかの情報を保存設定に記録。
   - 次回YMM4起動時に該当アプリが起動していれば自動で検知し、再アタッチを行う。
5. **安全な自動デタッチ (強制終了対策):**
   - YMM4終了時やパネル非表示時に、外部アプリケーションが非表示のままバックグラウンドに取り残されるのを防ぐ確実な解放＆原状復帰処理。

## 3. 実装方針と過去の開発から得られた知見（Must Read）

他のAI（Gemini, Claude等）へこのドキュメントを渡し、開発を進める際に必ず以下の構造と対策を守るように指示してください。

### 3.1 アタッチ手法 (SetParent方式)
- `NativeMethods.SetParent` を使用し、外部ウィンドウをWPFプラグイン内の `WindowsFormsHost` 上に乗せた `System.Windows.Forms.Panel` の子要素として埋め込む。
- 埋め込み時、`NativeMethods.GetWindowLong` / `SetWindowLong` を使用して対象の外部ウィンドウから `WS_CAPTION` (タイトルバー) や `WS_THICKFRAME` (サイズ変更枠) を削除し、パネルの枠へシームレスに統合する。
- **【注意】** WinUI 3などのモダンアプリ等では、`GWLP_HWNDPARENT`（所有権）を書き換えて無理矢理YMM4のウィンドウに従属させると、ウィンドウフォーカス喪失時に描画プロセスがフリーズし、画面が真っ白になるバグが確認されている。所有権の書き換えは行わず、純粋な `SetParent` やZオーダーでの制御を主軸に置くこと。

### 3.2 ドラッグ＆ドロップアタッチの検知
- 通常のWPFドラッグ＆ドロップイベントは、ドロップ対象が外部アプリケーションの別プロセスウィンドウであるためWPF側で検知できない。
- **【解決策】** `SetWinEventHook` を用いて、OS全体の `EVENT_SYSTEM_MOVESIZEEND` をフックすること。
- タイトルバーのドラッグ完了（ドロップ）が検知された際、`GetCursorPos` でマウス座標を取得し、それが該当パネル（WindowsFormsHost）の領域内であった場合、かつ `System.Windows.Input.Keyboard.IsKeyDown(Key.LeftCtrl)` 等でCtrlキーが押下されているかを判定して初めてアタッチ処理を実行する。

### 3.3 状態管理とタブ再生成への対策 (AvalonDock特有の振る舞い)
- YMM4が使用しているAvalonDockは、タブの再配置や別のタブへの切り替えが行われた時に、プラグインの `ViewModel` と `View` を躊躇なく破棄・再生成する性質がある。
- そのため、対象アプリの `HWND`（ウィンドウハンドル）や復元用のオリジナルStyle（`int OriginalStyle`）を、インスタンス変数の内部に普通に保持してしまうと、タブの再生成時に完全にロストし、二度と元のウィンドウに戻せなくなる。
- **【解決策】** 管理マネージャーやViewModelは、インデックスに対応した `AttachedHwnd` や `OriginalStyle` などを `static` な辞書や配列として、ライフサイクル全体で独立維持すること。

### 3.4 YMM4終了時の確実なデタッチ (コレクション変更エラーの罠対策)
- YMM4終了時にすべての外部アプリを確実に解放（デタッチ・独立ウィンドウ化）させるため、`Application.Current.MainWindow.Closing` および `AppDomain.CurrentDomain.ProcessExit` をフックして一括解放（FullDetachAll）を実行する。
- **【最重要ポイント（過去のバグ事例）】**
  解放アクション（Actionデリゲートのリスト）を `foreach` でループ実行する際、それぞれのアクション自体が「処理終了後にリストから自身を `Remove` する」ような自己完了型の登録解除処理を行っていた場合、**`foreach` ループ中にリストが改変されたとして `InvalidOperationException`** が生じ、ループがエラー停止する。
  すると2番目以降に登録されていた外部ウィンドウが解放されずバックグラウンドに取り残されるという重大なバグに発展する。
  解散アクションのループ実行は、**必ず対象リストを `.ToArray()` や `.ToList()` でコピーしたスナップショットを作り、それに対して回すこと。**

### 3.5 最大10個の動的パネル生成に関する、YMM4のアーキテクチャ制約
- YMM4のプラグイン仕様上、表示メニューに並ぶパネル群は `IToolPlugin` インターフェースを実装しており、プラグイン読み込み時にリフレクションで初期収集されて決定される。
- メニュー項目やドッキングパネル自体を完全な「実行時の動的生成」で行うAPIは明確に提供されていないケースが多い。
- **【安全で確実なアプローチ】** `GenericDockTargetPlugin01` ～ `GenericDockTargetPlugin10` のように、01〜10までを担う10個の汎用プラグインクラス群（ViewModelとView）をあらかじめ物理コードとして定義しておく。
- プログラムコードが冗長になるのを避ける場合、C#の Source Generator を利用するか、抽象基底クラスを用いて数行で10個定義するなどの工夫を取る。
- ドッキングUI管理画面パネルから、それらの対応インスタンスへ指示を出して対象アプリを管理するアーキテクチャが最も手堅い。
- 常に **10パネル分**（Dock 01〜10）のツールエントリを提供する。利用可否を設定で絞り込む必要はない。

- パネル名は、何かアタッチ状態になるアプリケーションがある場合、そのアプリケーションの名前を表示する。
- Windwos標準のエクスプローラーでタブを切り替えると、相性問題で表示できなくなるため、アタッチ不可とする。この際、アタッチできないことをメッセージで理由付きで表示する
